# Database migrations (Alembic)

SecuriSphere uses **Alembic** for schema versioning. Startup calls `migrate_schema()` â†’ `alembic upgrade head`.

## Revision chain

| Revision | File | Purpose |
|----------|------|---------|
| `001_baseline` | `001_baseline.py` | Create all tables from SQLAlchemy models |
| `002_indexes` | `002_indexes.py` | Performance indexes (idempotent) |
| `003_constraints` | `003_constraints.py` | CHECK constraints |
| `004_event_partitions` | `004_event_partitions.py` | Monthly RANGE partitions on `events` (skipped unless `EVENT_PARTITIONING_ENABLED=true`) |
| `005_agent_cert` | `005_agent_cert.py` | Agent mTLS fingerprint column |

**Note:** Keep `EVENT_PARTITIONING_ENABLED=false` for pilot/demo until the Event model and offense FKs are reconciled with composite partition keys.

## Commands (from `backend/`)

```bash
# Apply all migrations
alembic upgrade head

# Current revision
alembic current

# New autogenerated revision after model changes
alembic revision --autogenerate -m "describe_change"
```

Set `DATABASE_URL` in `.env` (async URL is fine; Alembic uses `psycopg2` sync driver internally).

## Existing databases (pre-Alembic)

If your database was created by the legacy `migrate.py` loops and has **no** `alembic_version` table, either:

1. **Let startup migrate** â€” first `alembic upgrade head` is idempotent (`CREATE IF NOT EXISTS`), or
2. **Stamp without running DDL** â€” if you are sure schema matches head:

```bash
alembic stamp head
```
